# まちAction Z世代向け価値設計 議事録

**日付：** 2026-07-01  
**形式：** マルチエージェント討議（企画 / マーケティング / 営業 / エンジニア + ファシリテーター）  
**テーマ：** Z世代に「毎日起動される」アプリにするための価値設計

---

## 1. Z世代へのコアインサイト（全エージェント一致）

### 表層インサイト：「得する」ではなく「選んだ自分を見せたい」

Z世代はクーポン金額に動かされるのではなく、**「3択の中から自分が選んだ」という自己決定の感覚と、それを誰かに見せたときの「自分らしさの表明」** に動かされる。ポイ活アプリとの最大の違いはここにある。

### 深層インサイト：FOMO × 社会貢献の二重刺激

「取り残されたくない」というFOMO感情と、「自分の行動が商店街を動かした」という貢献実感は同世代の中に共存している。まちActionの3択UIはこの両方を1回のアクションで充足できる構造を持っており、それが他のアプリにはない固有の価値である。

### 前提条件インサイト：「即・見える・共有できる」がなければ全部崩れる

タップしてから2〜3秒以内に反応が返らなければZ世代は離脱する。どれだけ優れた施策を積んでも、この即時性の前提条件が上流にある。

---

## 2.「選んだ自分を見せたい」への具体的な回答

エンジニア・マーケの対立を踏まえ、**3フェーズに段階分割** することで合意。

| フェーズ | 実装内容 | 見せ方 | 工数 |
|--------|---------|--------|------|
| **Phase 1（即日）** | React Native `Share.share()` でテキストシェア | 「今日、〇〇商店街を寄り道した。100円もらえた。 #まちAction」 | 即日 |
| **Phase 2（次スプリント）** | `experienceSummary` フィールド追加後にシェアカードUI（テキストベース） | 達成内容・報酬・ストリーク数をカード形式で表示 | 1〜2週間 |
| **Phase 3（将来）** | 達成ログの15秒サマリー画像自動生成・SNS直投稿 | Instagramストーリーズへの直接シェア | 要検討 |

### 非金銭的な自己表明の仕組み
- **「推しまちアンバサダー」称号**：同一エリアを10回以上達成したユーザーに付与。商店街の公式SNSからオプトイン型でシェア可能（義務化しない）。
- **達成者数の社会的証明**：「昨日の達成者 312人」のような数値をカードに表示し、FOMO感情を刺激。

## 3. 各エージェントの提案サマリー

### 企画エージェント
**インサイト：** Z世代はFOMO感情と社会貢献実感が同時に刺さる世代であり、まちActionはその両方を日常の寄り道体験で提供できる。

| 提案 | 優先度 |
|-----|------|
| 「今日の3択」をデイリーミッション化（残り時間カウントダウン・達成者数表示） | 高 |
| よりみちストリークで連続達成の快感ループを設計 | 高 |
| 「推しまち」UGCアンバサダー設計（称号・商店街SNS連携） | 中 |
| タイムウィンドウフィルタで「今の自分だけに出るカード」を演出 | 中 |

**懸念：** エンジニアへ → ゲーミフィケーション実装はバックエンドのスケーラビリティ問題を先に解決してから着手してほしい。

---

### マーケティングエージェント
**インサイト：** Z世代は「ブランドの言葉」ではなく「友達の行動」に動く。まちActionはUGCの仕組みを最初から設計に組み込まないとブランドが形成されない。

| 提案 | 優先度 |
|-----|------|
| テキストコピーの全面刷新（機能語→感情語）| 高 |
| 達成完了シェアカード＋`#まちAction` UGC設計 | 高 |
| 「スキマをデザインする」コンセプトでブランドポジショニング確立 | 中 |
| まちナビゲーター先行招待制度（コアZ世代ユーザーを口コミの核に） | 中 |

**懸念：** エンジニアへ → シェアカード実装のバックエンドサポートが必要。experienceSummaryフィールドを先に作ってほしい。

---

### 営業エージェント
**インサイト：** Z世代が「好き」なブランド（タピオカ・古着・コスメ・ゲーセン）を最初の広告主として口説くことで、ユーザーにとって「参加したいAction」が生まれる。

| 提案 | 優先度 |
|-----|------|
| Z世代に刺さる業態（古着・タピオカ・コスメ）を最初の広告主として獲得 | 高 |
| 大学QRポスター＋初回100円付与で最初の500人を大学生から獲得 | 高 |
| ストリーク常連バッジ→リアル店舗特典でBtoB営業トークを強化 | 中 |
| Plan Builder MVP（広告主セルフ入稿）で申し込み後の離脱を防ぐ | 中 |

**懸念：** 企画へ → 広告主が申し込んだ後の「Plan作成サポート」オペレーションが未設計。営業が拾う範囲を定義してほしい。

---

### エンジニアエージェント
**インサイト：** Z世代の離脱はUXの「待ち時間」ではなく「フィードバックがない時間」で発生する。即時フィードバック設計がすべての前提条件。

| 提案 | 優先度 |
|-----|------|
| アニメーションフィードバックの強化（Animated.spring によるバッジ獲得演出） | 高 |
| SQLite + FastAPI移行でスケーラビリティ基盤を整備 | 高 |
| プッシュ通知のパーソナライズ（タイムウィンドウ連動・ストリーク警告） | 中 |
| バックグラウンドロケーション追跡の最適化（iOS/Android 両対応） | 中 |

**懸念：** 企画・マーケへ → 現状プランが3本しかないため、既達成除外ロジックを入れると3日でカードが空になる。「常時7本以上 published」の運用ルールを先に定義してほしい。

## 4. 対立点と落としどころ

### ①「SQLite+FastAPI移行」はMVP段階で優先度「高」か
- **エンジニア：** 同時接続耐性に根本的な問題があり、先行投資として今やるべき
- **マーケ・営業：** MVP段階で100人同時接続は現実的でない。その工数をUXに使うべき
- **落としどころ：** ユーザー200人超えで無条件移行。それまでは書き込み競合が実測で発生したらSQLite優先に切り替えるトリガー基準をチームで事前合意する

### ②シェアカード・UGC設計を「優先度:高」に置くか
- **マーケ：** 最初から設計に組み込まないとブランドが形成されない
- **エンジニア：** 画像生成機能も experienceSummary もない現状では1スプリントに収まらない
- **落としどころ：** 3段階分割（Phase 1: テキストシェア即日 → Phase 2: カードUI → Phase 3: 画像生成）

### ③大学QRポスター施策の報酬原資問題
- **営業：** 100円初回付与で最初の500人を大学生から獲得したい
- **企画：** 広告主ゼロの段階で報酬原資を誰が負担するかが不明確
- **落としどころ：** 大学を「最初の広告主」として位置づける。学食・学内施設をActionスポットに加え、大学協定費を報酬原資に充当するスキームを組む

### ④「推しまち」アンバサダーの商店街SNSメンション設計
- **企画：** 商店街の公式SNSからメンションする設計を提案
- **営業：** 商店街側の手動オペレーションコストが発生し、契約後のクレームになる可能性
- **落としどころ：** 「商店街アカウントにシェアしませんか？」というオプトイン型にとどめ、商店街側にメンション義務を課さない

## 5. 次スプリントのエンジニアタスク（優先順）

1. **i18n文字列の更新**（マーケ主導でPR作成、エンジニアはマージのみ）1〜2時間
   - 「達成済み」→「寄り道、完了。」
   - 「クーポンを取得」→「今日の地元貢献が記録されました。」
   - 「報酬」→「今日のごほうび」

2. **`services.py` に `get_user_stats` 関数を追加**（2〜3日）
   - streak / total_completions / badges / plan_counts を返す
   - SQLite移行フラグを設計段階から意識して実装

3. **Action完了レスポンスに `experienceSummary` フィールドを追加**（1〜2日）
   - `planTitle / sponsorName / rewardAmount / durationMinutes / completedPlaceNames / streakCount`
   - フロント側のテキストシェア機能（Phase 1）も同時実装

4. **`list_plan_candidates` にタイムウィンドウフィルタ＋既達成除外ロジック追加**（1日）
   - 前提条件：企画チームが「常時7本以上 published」の運用データを用意してから着手

5. **Plan Builder MVP（広告主セルフ入稿Webフォーム）**（1週間）
   - 前提条件：営業チームがタピオカ・古着・コスメ3業態のヒアリングを完了し、APIフィールドとのギャップリストを提出してから着手

## 6. キャッチコピー案（3案）

| 案 | コピー | 用途 | 採択 |
|----|-------|------|------|
| **A（FOMO × 日常）** | 「帰り道は、まだ終わってない。」 | 通知文・TikTokキャプション | ✅ **採択** |
| B（自己決定 × 社会貢献） | 「今日の3択が、まちを少し動かした。」 | 達成完了画面・シェアカード | 保留 |
| C（タイパ × ライフスタイル） | 「スキマを、寄り道に変える。」 | ストア説明・オンボーディング | 保留 |

> **採択理由（2026-07-01）：** 通知文・TikTokキャプションはケースAを使う。FOMO感情に直接刺さる表現でZ世代の帰り道という日常の場面に最もフィットする。

---

## 7. ファシリテーターの最終所見

> 次のスプリントでチーム全体が合意すべき最小単位は「**毎朝アプリを開く理由が1つあること**」だけだ。ストリーク表示でも、コピー刷新でも何でもよい。だが1つを選んで完成させ、ユーザーの実際の行動データを取ってから次を決める。この順序を守ることがまちActionをMVPで終わらせないための唯一の方法だ。
